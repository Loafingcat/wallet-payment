# Transactional Outbox 패턴 (S10)

S6(커밋 `983c4d5`)에서 `@TransactionalEventListener(AFTER_COMMIT)`로 "결제 트랜잭션이 commit된 후에만 RabbitMQ에 발행"하도록 만들었다. 이건 "커밋 전에 발행해서 롤백된 거래가 발행되는" 문제는 막아주지만, **다른 방향의 문제**가 남아있었다: **커밋은 됐는데, 그 직후 발행이 실패하거나 그 직전에 프로세스가 죽으면 이벤트가 영원히 사라진다.** DB에는 결제가 성공했다는 기록(LedgerEntry, Wallet.balance)이 분명히 남아있는데, "이 결제에 대해 이벤트를 발행해야 한다"는 사실은 메모리에서만 잠깐 존재했다가 사라진다 — 어디에도 다시 시도할 근거가 안 남는다.

## 핵심 아이디어

"비즈니스 데이터 변경"과 "발행할 이벤트가 있다"는 사실을 **같은 로컬 DB 트랜잭션**에 같이 커밋한다. 이러면 이 둘은 같은 원자성을 가진다 — 트랜잭션이 성공하면 둘 다 있고, 실패하면 둘 다 없다. RabbitMQ로의 실제 발행은 그 트랜잭션이 끝난 *후*, 별도의 릴레이(`OutboxRelay`)가 outbox 테이블의 `PENDING` 행을 읽어서 처리한다. 릴레이가 언제 죽든, 몇 번 재시작하든, `PENDING` 행은 DB에 그대로 남아있으므로 다시 시도하면 그만이다.

```
[PaymentService.pay() 트랜잭션]
  wallet.pay(amount)
  LedgerEntry INSERT          ─┐
  OutboxEvent INSERT(PENDING) ─┘  같은 트랜잭션, 같은 커밋

[OutboxRelay, 2초마다 폴링 — 트랜잭션 밖]
  PENDING 조회 → RabbitMQ 발행 → 성공하면 PUBLISHED로 표시
```

## 기존 방식(@TransactionalEventListener) vs 아웃박스

| | `@TransactionalEventListener(AFTER_COMMIT)` (S6) | Outbox + Relay (S10) |
|---|---|---|
| "발행해야 한다"는 사실이 어디에 남는가 | 어디에도 안 남음 — JVM 메모리의 이벤트 리스너 호출 스택에만 잠깐 존재 | `outbox_event` 테이블에 비즈니스 데이터와 **같은 트랜잭션으로** 영속화됨 |
| 커밋 후, 발행 *직전*에 프로세스가 죽으면 | **이벤트 영구 유실.** 재시작해도 "이런 이벤트가 있었다"는 걸 알 방법이 없다 | **유실 없음.** 재시작한 릴레이가 PENDING 행을 그대로 찾아서 발행한다 |
| 발행 시도 자체가 예외로 실패하면 | 그 자리에서 끝 — 재시도 로직이 없다 | PENDING으로 남아 다음 폴링(2초 후)에서 다시 시도된다 |
| 발행 시점 | 커밋 직후, 같은 요청을 처리하던 스레드 안에서 즉시(동기) | 폴링 주기만큼 지연(이 프로젝트는 최대 2초) |
| 보장 수준 | "커밋되면 발행을 *시도*한다" (best-effort, 그 시도가 끝까지 성공한다는 보장은 없음) | "커밋되면 *언젠가는 반드시* 발행된다" (at-least-once) |
| 중복 발행 가능성 | 거의 없음(한 번만 시도, 재시도가 없으니 중복도 없음 — 그 대신 유실 위험을 안고 있음) | 있음 — 발행에는 성공했는데 `PUBLISHED`로 표시하기 *전*에 릴레이가 죽으면, 다음 폴링이 같은 행을 또 보낸다 |
| 구현 복잡도 | 단순(리스너 메서드 하나) | 테이블 + 릴레이 + 스케줄러 필요 |

요약하면: **AFTER_COMMIT은 유실 위험을 안고 빠르고 단순하며, 아웃박스는 유실을 없애는 대신 약간의 지연과 중복 발행 가능성을 받아들인다.** 중복 발행은 막을 수 없는 게 아니라 "막을 필요가 없는" 문제다 — 멱등성 있는 소비자라면 중복이 와도 안전하기 때문이다.

## 소비자 멱등성을 "유지"한 이유 (강화하지 않은 이유)

`PaymentNotificationListener`(S6)는 이미 `ProcessedPaymentEvent`(paymentId가 PK)로 같은 결제완료 이벤트를 두 번 처리하지 않도록 막고 있었다. 아웃박스가 도입하는 "중복 발행" 가능성도 **소비자 입장에서는 S6 시절과 똑같은 모양의 문제**(같은 메시지가 두 번 옴)라서, 이미 있는 멱등성 처리가 그대로 막아준다 — 새로 코드를 추가할 필요가 없었다. `OutboxRelayTest`의 `같은_이벤트가_중복_발행돼도_소비자_멱등성이_중복_처리를_막는다()`로 이걸 직접 재현해서 확인했다(릴레이를 거치지 않고 같은 이벤트를 두 번 직접 발행해, "발행 성공 후 PUBLISHED 표시 전에 릴레이가 죽는" 상황을 흉내냈다).

## 유실 0 보장 테스트

`OutboxRelayTest`가 검증하는 것:

1. **결제_트랜잭션이_커밋되면_발행_전이라도_outbox에_안전하게_남는다**: 결제가 끝난 직후(릴레이가 한 번도 안 돈 시점)에도 지갑/원장은 이미 확정돼 있고, outbox 행은 `PENDING`으로 존재한다.
2. **재구동을_시뮬레이션한_릴레이_실행은_유실_없이_PENDING_이벤트를_발행한다**: "프로세스가 죽었다가 재시작했다"를 흉내내려고 JVM을 실제로 죽이는 대신, 그냥 릴레이를 처음 호출한다 — 릴레이 입장에서는 "이 PENDING 행이 방금 생긴 건지, 1분 전에 생긴 건지" 구분할 필요도 이유도 없다. 그게 이 패턴의 핵심이다: **상태가 DB에 있어서, "왜 아직 안 보냈는지"를 몰라도 안전하게 다시 시도할 수 있다.**
3. **릴레이가_발행한_메시지를_소비자가_정상_처리한다**: 끝까지 실제로 RabbitMQ를 거쳐 컨슈머가 처리하는지 확인.
4. **알수없는_이벤트_타입이_섞여도_나머지_PENDING_이벤트_발행은_계속된다**: 한 건이 망가져도(잘못된 eventType) 다른 PENDING 행 처리를 막지 않는다 — 그 행은 그대로 PENDING으로 남아 계속 재시도 대상이 된다(무한 재시도가 데이터 정합성을 해치진 않지만, 운영에서는 로그/알림으로 감지해야 할 부분이다).
5. **같은_이벤트가_중복_발행돼도_소비자_멱등성이_중복_처리를_막는다**: 위에서 설명한 시나리오.

## 의도적으로 안 한 것

- **outbox 행을 발행 후 삭제하지 않는다.** `PUBLISHED`로 표시만 하고 남겨둔다 — 감사(audit)와 디버깅(이 이벤트가 언제 발행됐는지)을 위해서다. 운영에서는 오래된 `PUBLISHED` 행을 주기적으로 청소하는 배치가 필요하겠지만, 이번 스코프에는 넣지 않았다.
- **재시도 횟수 제한이나 데드레터를 두지 않았다.** PENDING 행은 영원히 재시도된다. 진짜 운영 환경이라면 N번 실패한 행을 별도로 표시해 알림을 보내는 게 맞지만, 지금은 "유실 0"이라는 핵심 보장을 보여주는 데 집중했다.
