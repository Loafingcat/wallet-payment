# 분산 정합성: 외부 PG 연동 (S9)

지금까지(S1~S8)는 "내 DB만 바꾸면 끝나는" 문제였다. 지갑, 원장, 결제, 환불, 정산 — 전부 우리 MySQL 하나 안에서 트랜잭션 하나로 끝났다. S9부터는 다르다: 결제가 성립하려면 **우리가 통제할 수 없는 외부 시스템(PG)**의 승인이 필요하다. 이 문서는 "우리 DB"와 "PG"라는 두 개의 독립된 시스템 사이에서 발생하는 정합성 문제를 어떻게 풀었는지 설명한다.

## 핵심 질문

> 우리 DB 커밋은 됐는데 PG 호출이 실패하면? (반대도)

이 질문에 깔린 가정이 이미 틀렸다 — **"우리 DB 커밋"과 "PG 호출"은 절대 같은 트랜잭션에 들어갈 수 없다.** PG는 우리 코드가 아니라 HTTP로만 닿을 수 있는 남의 시스템이다. 그래서 정확한 질문은 이거다:

> PG를 호출했는데, 그 호출이 **성공/명확한 실패/모름(타임아웃, 응답유실)** 중 무엇인지조차 우리가 확신할 수 없는 순간이 있다. 그 순간, 우리 DB 상태를 어떻게 둬야 하는가?

## 왜 2PC로 풀지 않는가

2PC(Two-Phase Commit)는 모든 참여자가 "준비(prepare)" 상태로 대기하다가 코디네이터의 신호에 맞춰 다같이 commit하는 프로토콜이다. 이게 PG 연동에는 구조적으로 안 맞는다:

1. **PG는 우리 트랜잭션의 참여자가 될 수 없다.** 2PC 참여자가 되려면 "준비됐다"는 상태로 잠금을 걸고 대기해야 하는데, PG는 우리 말고도 수많은 다른 가맹점의 요청을 처리하는 독립된 서비스다. 우리 코디네이터의 신호를 기다리며 자기 자원을 잠가줄 이유가 없고, 애초에 그런 프로토콜(XA 등)을 제공하는 PG도 없다.
2. **블로킹 프로토콜이라 네트워크 장애에 취약하다.** 코디네이터(우리)가 prepare 이후 commit 신호를 보내기 전에 죽거나 네트워크가 끊기면, 참여자는 잠금을 풀지도 못하고 영원히 대기할 수 있다(blocking). 이 프로젝트가 정확히 재현하는 장애(타임아웃, 응답유실)가 2PC를 가장 무력하게 만드는 상황이다.
3. **PG가 동기적으로 우리를 기다려줄 이유가 없다.** 실제 PG는 자기 SLA에 맞춰 빨리 응답하고 다음 요청을 처리해야 한다 — 우리가 "2단계 commit 신호"를 보낼 때까지 자원을 묶어두는 건 PG 입장에서 받아들일 수 없는 설계다.

## 대신 쓴 방법: 멱등성 + 상태기계 + 보정

**멱등성**: PG 호출은 언제든 같은 키로 다시 보내도 안전하다(PG가 직접 멱등키를 지원하도록 시뮬레이션했다). 그래서 "성공했는지 모르겠다"는 상황에서, 정답을 모르면 그냥 **나중에 같은 키로 다시 물어보거나 다시 시도**하면 된다 — 중복 승인 걱정 없이.

**상태기계**: 결제에 `PENDING_PG`라는 "모른다"는 상태를 명시적으로 둔다. 이게 핵심이다 — 흔한 실수는 "성공 응답을 못 받았다"를 "실패"로 합쳐버리는 것인데, 그러면 사실 PG가 승인한 결제를 우리가 실패로 처리해서 사용자에게 손해를 입히거나(돈은 나갔는데 우리 쪽엔 기록이 없음), 반대로 안전을 위해 매번 돈을 움직이지 않는 쪽으로 보수적으로 가면 사용자가 "결제했는데 왜 안 되냐"고 느끼는 경험이 나빠진다. `PENDING_PG`는 이 모호함을 숨기지 않고 그대로 데이터에 남긴다.

**보정(reconciliation)**: 모르는 상태(`PENDING_PG`)를 영원히 모른 채로 두지 않고, 시간이 지난 뒤 PG에 직접 물어봐서(`GET /pg/payments/{key}`) 진실을 확인한다. 이게 가능한 이유는 **PG가 우리와 무관하게 자기 자신과는 항상 정합성을 유지하기 때문**이다 — PG는 처리했으면 했고, 안 했으면 안 한 거고, 그 사실을 나중에라도 물어보면 정확히 답해준다. 우리는 그 답이 도착하기를 기다리는 대신, 그 사이엔 돈을 움직이지 않고(`PENDING_PG`로 유보), 답을 알게 된 시점에 확정한다.

이 셋을 합치면 **"즉시 일관성"은 포기하지만 "최종 일관성"은 보장**된다 — 그리고 그 사이 어떤 순간에도 절대 규칙 1번(잔액 음수 불가)과 2번(원장 = 진실의 원천)은 깨지지 않는다. 돈은 **PG가 승인했다는 게 확인된 후에만** 움직이기 때문이다.

## 상태 다이어그램

```
                 POST /payments/external
                          │
                          ▼
                  ┌───────────────┐
                  │  PENDING_PG   │  ◄── (멱등) 같은 키로 재요청 와도 PG 또 호출
                  └───────────────┘
                    │      │      │
        PG 응답:    │      │      │   PG 호출 결과: 모름
        승인        │      │      │   (타임아웃 / 응답유실 / 커넥션 오류)
                    │      │      └──────────────┐
                    │      │                     │ (그대로 PENDING_PG 유지)
                    │      │ PG 응답: 명확한 거부  │
                    │      │ (5xx)                │
                    ▼      ▼                      ▼
            ┌───────────┐ ┌──────────┐    [보정: GET /pg/payments/{key}]
            │ APPROVED  │ │ FAILED   │             │
            │(LedgerEntry│ └──────────┘     ┌───────┼────────┐
            │  기록됨)   │                  │       │        │
            └───────────┘            PG: APPROVED  PG: NOT_FOUND   PG: 조회도 실패
                  ▲                        │     (+유예기간 경과)    (UNKNOWN)
                  │                        │              │            │
                  └────────────────────────┘              ▼            │
                                                      ┌──────────┐      │
                                                      │ FAILED   │      │
                                                      └──────────┘      │
                                                                        ▼
                                                              (PENDING_PG 유지,
                                                               다음 스케줄에서 재시도)
```

`PENDING_PG`에서 나가는 화살표는 두 군데서 출발할 수 있다 — 최초 호출(`ExternalPaymentService.requestPayment`)과 보정(`PaymentReconciliationService.reconcileOne`). 둘 다 결국 같은 두 메서드(`ExternalPaymentWriter.confirmApproved` / `markFailed`)로 수렴하고, 그 메서드들은 `payment.isPending()`을 다시 확인한 뒤에만 동작한다 — 그래서 최초 호출과 보정이 거의 동시에 같은 결제를 건드려도 중복 적용되지 않는다.

## 장애 케이스별 처리 (fake-pg의 `X-Simulate-Failure` 헤더)

| 시나리오 | PG 쪽 실제 상태 | 우리 쪽 1차 응답 | 보정 결과 |
|---|---|---|---|
| `none`(정상) | 즉시 승인 기록 | `APPROVED` (즉시) | — |
| `error5xx` | 승인 기록 안 함(명확한 실패) | `FAILED` (즉시) | — |
| `timeout` | 우리가 포기한 시점엔 처리 전, 곧 승인됨 | `PENDING_PG` | 유예 기간 안엔 그대로, 지연 끝나면 `APPROVED` |
| `lost-response` | 응답을 보내기 **전**에 이미 승인 기록 | `PENDING_PG` | 즉시 `APPROVED` (유예 기간 무관) |
| (PG에 끝내 도달 못함, 시뮬레이션 없이 직접 재현) | 영원히 기록 없음 | `PENDING_PG` | 유예 기간 경과 후 `FAILED` |

`timeout`과 `lost-response`는 우리 클라이언트 입장에선 증상이 똑같다(응답을 못 받음) — 그런데 **PG 쪽 진실은 다르다**(아직 처리 전 vs 이미 처리 완료). 이게 분산 시스템에서 가장 까다로운 지점이다: 같은 증상이 서로 다른 원인을 가질 수 있고, 클라이언트는 그 둘을 구분할 방법이 없다. 그래서 둘 다 일단 `PENDING_PG`로 같이 묶어두고, 보정이 PG에 직접 물어봐서(`GET /pg/payments/{key}`) 사후적으로 진실을 가려낸다 — 클라이언트가 "추측"하지 않고 "확인"한다는 게 핵심이다.

## 코드에서 절대 규칙 6번이 적용된 자리

`ExternalPaymentService.requestPayment()`는 `@Transactional`이 아니다. PG 호출(`PgClient.approve`)은 최대 1초까지 걸릴 수 있는(설정한 타임아웃) 네트워크 호출인데, 이걸 트랜잭션 안에 두면 그 시간 동안 DB 커넥션과 (경우에 따라) 지갑 행 잠금을 붙들고 있게 된다. 그래서 DB 쓰기는 `ExternalPaymentWriter`의 짧고 확정적인 트랜잭션 세 개(`createPending`/`confirmApproved`/`markFailed`)로 쪼개고, 오케스트레이터(`ExternalPaymentService`)는 그 사이에 트랜잭션 없이 PG를 호출한다. S6에서 RabbitMQ 발행에 적용했던 원칙과 똑같다 — 그때는 "발행을 커밋 후로 미룬다"였고, 여기는 "호출을 트랜잭션 밖으로 뺀다"인데 동기는 같다: **느리거나 불확실한 외부 작업이 DB 트랜잭션의 생사를 좌우하게 두지 않는다.**
