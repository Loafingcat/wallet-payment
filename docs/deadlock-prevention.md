# 송금 데드락: 재현과 해결 (S11)

S2에서 비관적 락(`PESSIMISTIC_WRITE`)으로 동시 결제 문제를 풀었지만, 그건 **지갑을 하나만** 잠그는 경우였다. 지갑 간 송금(`TransferService`)은 출금 지갑과 입금 지갑, **두 개를 한 트랜잭션에서 동시에** 잠가야 한다 — 그러자 비관적 락 자체가 새로운 문제를 만들 수 있다는 게 드러났다: **데드락**.

## 왜 데드락이 나는가

`TransferService.transfer(fromWalletId, toWalletId, ...)`의 첫 버전(커밋 `5c4c23a`)은 인자로 받은 순서 그대로 잠궜다 — `fromWalletId` 먼저, `toWalletId` 다음.

지갑 A(id=1), B(id=2)가 있다고 하자. 두 송금이 정확히 같은 시점에 들어온다:

| 시간 | 트랜잭션1 (A→B) | 트랜잭션2 (B→A) |
|---|---|---|
| t1 | A 잠금 성공 | B 잠금 성공 |
| t2 | B 잠금 시도 → **대기**(트랜잭션2가 들고 있음) | A 잠금 시도 → **대기**(트랜잭션1이 들고 있음) |

트랜잭션1은 "내가 가진 A를 풀어줄 생각이 없는" 트랜잭션2가 쥔 B를 기다리고, 트랜잭션2는 정반대로 트랜잭션1이 쥔 A를 기다린다. 둘 다 상대가 먼저 끝나야 풀려나는데, 상대는 자신이 끝나야 풀려나니, **누구도 먼저 끝날 수 없다** — 순환 대기(circular wait). MySQL InnoDB는 이 상태를 감지해서 둘 중 하나를 강제로 롤백시킨다(데드락 희생자, 에러 1213).

이건 타이밍에 의존하는 문제라서, 한 쌍만 동시에 보내면 안 걸릴 수도 있다(한쪽이 먼저 끝나버리면 애초에 경합이 안 생긴다). `TransferDeadlockTest`(S11-1, 커밋 `5c4c23a`)는 이를 신뢰성 있게 재현하기 위해 A→B/B→A 양방향으로 **10쌍을 동시에** 쏴서, 적어도 한 쌍이 겹치는 타이밍 창에 들어가도록 만들었다. 3회 연속 실행에서 매번 `PessimisticLockingFailureException`(MySQL 에러 1213이 Spring을 거쳐 변환된 예외)을 확인했다.

## 어떻게 고치는가: 락 순서 고정

해결책은 단순하다 — **두 지갑 중 id가 더 작은 쪽을 항상 먼저 잠근다**(커밋 `b56604e`). A→B든 B→A든 상관없이:

```java
Long firstLockId = Math.min(fromWalletId, toWalletId);
Long secondLockId = Math.max(fromWalletId, toWalletId);
```

이렇게 하면 같은 지갑 쌍(A, B)을 동시에 건드리는 **모든** 트랜잭션이 항상 "A 먼저, B 다음" 순서로만 락을 요청한다. 위 표를 다시 그려보면:

| 시간 | 트랜잭션1 (A→B) | 트랜잭션2 (B→A) |
|---|---|---|
| t1 | A 잠금 성공 | A 잠금 시도 → **대기** |
| t2 | B 잠금 성공 → 처리 → 커밋(A, B 둘 다 해제) | (계속 대기 중) |
| t3 | — | A 잠금 성공 → B 잠금 성공 → 처리 |

트랜잭션2는 트랜잭션1이 들고 있는 A를 기다리는 동안, 트랜잭션1이 갖지 않은 무언가를 트랜잭션2가 쥐고 있을 일이 없다 — 트랜잭션2는 아직 아무 락도 안 잡은 상태로 대기하기 때문이다. 순환 대기의 구조적 조건(서로가 서로의 다음 락을 쥐고 있음) 자체가 성립하지 않는다. 이건 *우연히* 안 걸리는 게 아니라 **수학적으로 걸릴 수 없는** 상태다 — 표준적인 데드락 방지 기법인 "전역 락 순서(lock ordering)"다.

`TransferDeadlockTest`는 이 수정 이후 정반대를 증명한다: 같은 부하(10쌍 동시)에서 데드락이 **전혀** 발생하지 않고, 모든 송금이 끝까지 성공하며, A→B와 B→A가 같은 횟수만큼 상쇄돼서 양쪽 지갑 잔액이 원래 값으로 돌아온다.

## 그래도 남는 문제: 단순 대기

락 순서를 고정해도 없어지지 않는 문제가 하나 있다 — **데드락이 아니라 단순 대기**다. 트랜잭션2가 A를 기다리는 동안, 트랜잭션1이 어떤 이유로든(버그, 매우 큰 배치, 네트워크 지연) 오래 안 끝나면 트랜잭션2는 그만큼 오래 멈춰 있는다. 데드락처럼 영원히 풀리지 않는 건 아니지만, MySQL 기본값(`innodb_lock_wait_timeout` = 50초)까지 사실상 무한정 기다리게 둘 수는 없다.

그래서 `TransferService.transfer()`는 락을 잡기 직전에 **자기 트랜잭션이 쓰는 세션에서만** 이 타임아웃을 3초로 줄인다:

```java
entityManager.createNativeQuery("SET innodb_lock_wait_timeout = 3").executeUpdate();
```

`SET innodb_lock_wait_timeout`은 세션 변수라서, 다른 트랜잭션/세션에는 전혀 영향을 주지 않는다 — 이 송금 하나만 "3초 넘게 기다리면 포기"하게 만든다. 3초를 넘기면 MySQL 에러 1205(lock wait timeout)가 나는데, 이것도 Spring이 `PessimisticLockingFailureException`으로 변환해준다(에러 1213 데드락과 같은 처리 경로를 탄다) — 호출하는 쪽(컨트롤러)은 "락을 못 잡았다"는 한 종류의 예외만 처리하면 된다.

**처음엔 JPA 표준 힌트(`jakarta.persistence.lock.timeout`)를 `@QueryHints`로 시도했었다.** `TransferLockTimeoutTest`를 작성해 직접 검증해보니 이 힌트는 Hibernate의 MySQL 다이얼렉트에서 실제로 적용되지 않았다(대기가 3초가 아니라 10초 넘게, 즉 테스트가 강제로 풀어줄 때까지 그대로 이어졌다) — 그래서 세션 변수를 직접 설정하는 방식으로 바꿨다. JPA 힌트의 실제 동작은 데이터베이스/드라이버 조합에 따라 다르므로, "표준 API니까 동작할 것"이라고 가정하지 않고 테스트로 직접 확인하는 게 중요하다는 걸 보여주는 사례다.

## 테스트

- **`TransferDeadlockTest`**: 락 순서 고정 전(이전 버전, `git show 5c4c23a:.../TransferDeadlockTest.java`로 확인 가능)엔 데드락이 발생함을, 고정 후(현재)엔 같은 부하에서 데드락이 발생하지 않고 모든 송금이 성공함을 증명.
- **`TransferLockTimeoutTest`**: 한 스레드가 지갑 락을 일부러 오래 들고 있을 때, 다른 스레드의 송금이 MySQL 기본값(50초)까지 기다리지 않고 짧게(3초 근처) 타임아웃되는지 확인.
- **`TransferServiceTest`**: 정상 송금, 멱등키 재요청, 자가송금 거부, 잔액 부족 시 양쪽 지갑 모두 변경되지 않는지 검증.
